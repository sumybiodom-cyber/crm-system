# CRM для сервисного центра (финальная версия под бизнес)

## 🎯 Цель
Создать простую, быструю и понятную CRM систему для сервисного центра:
- контроль заказов
- контроль денег
- контроль сотрудников
- минимальный интерфейс без перегруза

---

## 👥 Роли

### Менеджер
- приём клиента
- создание заказа
- фиксация проблемы
- согласование цены
- приём оплаты
- выдача заказа

---

### Инженер
- видит только свои заказы
- выполняет ремонт
- указывает работы
- добавляет запчасти
- переводит заказ по этапам

---

### Руководитель
- видит всё
- видит прибыль
- видит загрузку
- контролирует сотрудников

---

## 📦 Жизненный цикл заказа (СТРОГО)

## ЭКРАН РУКОВОДИТЕЛЯ — ДАШБОРД БИЗНЕСА

Цель: руководитель должен за 30 секунд понять, что происходит в сервисе.

### 1. Верхние показатели
Показать 4 карточки:

- Заказы сегодня
- В ремонте
- Готово к выдаче
- Долги клиентов

### 2. Деньги
Показать:

- Выручка сегодня
- Выручка за месяц
- Прибыль за месяц
- Сумма неоплаченных заказов

### 3. Контроль проблем
Показать только важное:

- Зависли в ремонте
- Готово, но не забрали
- Нет оплаты
- Ожидает согласования

### 4. Инженеры
Показать по каждому инженеру:

- сколько заказов в работе
- сколько готово
- сколько зависло
- сумма заработка инженера

### 5. Быстрые действия
Кнопки:

- Открыть все заказы
- Открыть готовые к выдаче
- Открыть долги
- Открыть зависшие

### 6. Главное правило
Экран руководителя не должен быть перегружен.

Не показывать:
- лишние графики
- сложную бухгалтерию
- мелкие технические детали

Показывать только:
- деньги
- заказы
- проблемы
- сотрудников

### 7. Суть
Руководитель должен видеть не всю CRM, а только контроль бизнеса.

## КОНТРОЛЬ ОШИБОК СОТРУДНИКОВ

Цель: система должна не мешать работе, а защищать от грубых ошибок менеджера и инженера.

### 1. Ошибки менеджера

Система должна предупреждать или блокировать, если:

- заказ создаётся без телефона клиента
- заказ создаётся без устройства
- заказ создаётся без описания проблемы
- заказ переводится в работу без назначенного инженера
- заказ выдаётся без оплаты или с долгом
- заказ готов, но клиент не уведомлён
- заказ закрывается без документа

### 2. Ошибки инженера

Система должна предупреждать или блокировать, если:

- заказ переводится в “Готово” без диагностики
- не указано, что было сделано
- не отмечено “проверено”
- использованы запчасти, но они не списаны
- заказ долго не менял статус

### 3. Уровни контроля

Жёстко блокировать:
- нет телефона
- нет устройства
- нет проблемы
- нет инженера при переводе в работу
- выдача без суммы

Только предупреждать:
- есть долг
- клиент не уведомлён
- заказ завис
- нет ячейки хранения
- похожий заказ уже существует

### 4. Журнал действий

Если сотрудник подтверждает рискованное действие, система должна записать:

- кто сделал
- когда
- какое действие
- какой был риск

Пример:
“Менеджер выдал заказ с остатком долга 500 грн.”

### 5. Главное правило

Система не должна быть сложной.  
Она должна ловить только ошибки, которые приводят к потере денег, техники или клиента.
