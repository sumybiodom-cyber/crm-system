# Структура бази даних CRM СПЕКТР-АС

Файл схеми: [`db/schema.sql`](../db/schema.sql)

Схема розрахована на PostgreSQL і підготовлена для backend-рівня CRM/ERP: ремонти, продажі, склад, закупки, оплати, ролі, аудит і аналітика.

## Основні принципи

- `id` у таблицях використовує `uuid`.
- Критичні зв'язки мають `foreign key`.
- Документи не видаляються фізично, де потрібна історія: використовується `deleted_at`.
- Майже всі довідники та документи мають `created_at` і `updated_at`.
- Склад змінюється тільки через `stock_movements`.
- Оплата не списує товар. Списання відбувається через документ видачі/закриття.
- Критичні дії пишуться в `audit_logs`.

## Блоки таблиць

### Користувачі та права

- `users`
- `roles`
- `permissions`
- `role_permissions`

Цей блок відповідає за логін, ролі, точкові права, активність користувача, 2FA-прапорець і примусовий вихід.

### Клієнти, постачальники, товари

- `clients`
- `suppliers`
- `warehouses`
- `product_categories`
- `products`
- `product_locations`
- `product_prices`

Тут зберігаються довідники для ремонтів, продажів, складу і закупівель.

У `products.barcode` зберігається штрих-код товару. По ньому є індекс, щоб USB-сканер міг швидко знаходити товар у продажі, складі або приході.

### Склад

- `stock_balances`
- `stock_movements`
- `inventory_sessions`
- `inventory_session_items`

`stock_balances.qty_available` генерується як:

```text
qty_on_hand - qty_reserved
```

Тому доступний залишок не потрібно рахувати вручну в кожному запиті.

### Сервіс і ремонти

- `service_orders`
- `service_order_status_history`
- `service_order_assignments`
- `service_order_items`
- `service_part_requirements`
- `service_parts_usage`
- `order_groups`
- `order_units`
- `order_unit_movements`
- `storage_cells`

Потреба в деталі, її закупка, резерв, видача інженеру, встановлення і списання мають бути пов'язані з конкретним `service_order_id`.

`service_order_items.engineer_id` фіксує, хто виконав конкретну роботу. Це основа для зарплати інженерів.

Кожна строка роботи має власну схему нарахування:

- `service_type` — тип послуги, наприклад ремонт, заправка картриджа, регенерація картриджа;
- `payroll_accrual_type` — `percent_of_work_amount` або `fixed_per_unit`;
- `payroll_percent_rate` — процент для звичайного ремонту;
- `payroll_fixed_per_unit` — ставка за одиницю для заправки або регенерації;
- `payroll_accrual_amount` — зафіксована сума нарахування;
- `payroll_locked_at` — момент блокування після видачі або закриття.

У `service_orders` суми ремонту зберігаються окремо:

- `total_work_amount` — сума робіт;
- `total_parts_amount` — сума запчастин;
- `delivery_amount` — доставка;
- `total_amount` — загальна сума.

Для зарплати інженера використовуються тільки строки `service_order_items`. Запчастини, витратні матеріали, доставка і додаткові витрати не входять у базу зарплати, якщо вони не оформлені окремою послугою.

`order_groups` звʼязує кілька заказів одного клієнта в одну передачу. Наприклад: окремий заказ на заправку і окремий заказ на регенерацію картриджів.

`order_units` зберігає кожну фізичну одиницю: пристрій або картридж. Кожна одиниця має унікальний код, QR/штрихкод, статус, інженера і місце зберігання.

`storage_cells` зберігає координатні комірки формату `Shelf-Level-Position`, наприклад `A-3-2`. Якщо одиниця має статус `on_shelf`, `storage_cell_id` обовʼязковий на рівні бізнес-логіки.

`order_unit_movements` фіксує рух по цеху: хто змінив статус, коли, куди поклав, який статус був і який став.

Для QR-коду замовлення використовуються:

- `service_orders.qr_token`;
- `service_orders.qr_url`.

QR має відкривати картку замовлення напряму. У frontend-MVP це посилання виду `/orders/номер`, у backend-версії воно має вести на захищений маршрут з перевіркою доступу.

`service_orders.status` описує окремий маршрут ремонту:

```text
accepted -> diagnostics -> awaiting_approval -> approval -> waiting_parts / in_progress -> testing -> ready -> awaiting_payment -> issued -> closed
```

Альтернативні статуси:

- `cancelled`
- `cannot_repair`
- `waiting_client`
- `paused`
- `warranty_return`

Статус оплати ремонту зберігається окремо в `service_orders.payment_status`:

- `unpaid`
- `prepayment`
- `partially_paid`
- `fully_paid`
- `refund`
- `debt`

Це важливо: ремонт може бути `ready`, а оплата при цьому `partially_paid`.

Кожна зміна статусу записується в `service_order_status_history`.

`cannot_repair` у frontend відображається як `Не підлягає ремонту`. Статус може встановити інженер або менеджер, обовʼязково з коментарем-причиною. Після цього заказ не рахується активним ремонтом і переходить у список готових до видачі клієнту.

Кожне призначення або переназначення інженера записується в `service_order_assignments`.

Правило розподілу:

- один фізичний пристрій = один `service_order`;
- один клієнт може мати багато `service_orders`;
- інженер бачить тільки замовлення, де він вказаний у `engineer_id`;
- інженер має підтвердити роботу через `engineer_accepted_at`;
- без `engineer_id` і `engineer_accepted_at` backend не повинен дозволяти статус `in_progress`;
- замовлення інших інженерів не повинні потрапляти у відповідь API для ролі інженера;
- кожна зміна призначення, підтвердження і статусу пишеться в `audit_logs`;
- керівник і адміністратор бачать усі замовлення та навантаження інженерів.

- сума з ПДВ;

### Зарплата

- `payroll_rules`
- `payroll_periods`
- `payroll_accruals`

`payroll_rules` зберігає базові правила оплати співробітників. Для інженера головне поле:

- `percent_rate` — індивідуальний процент від суми робіт.

Формула:

```text
percent_of_work_amount = percent_rate × service_order_items.total_price
fixed_per_unit = fixed_per_unit × service_order_items.qty
```

Фіксована ставка за одиницю потрібна для робіт типу заправки картриджів або регенерації. Якщо в одному заказі є різні типи робіт, кожна строка рахується окремо за своєю схемою, а потім сума агрегується по інженеру.

`payroll_periods` зберігає період розрахунку: чернетка, розраховано, затверджено, виплачено.

`payroll_accruals` зберігає нарахування по конкретних роботах із `service_order_items`. Поля `accrual_type`, `percent_rate`, `fixed_per_unit`, `work_amount` і `accrual_amount` фіксують саме те правило, за яким була розрахована зарплата.

Головне правило: зарплата інженера рахується не від заказу цілком і не від оплати клієнта, а від виконаних робіт. Нарахування створюються тільки після статусу ремонту `issued` або `closed`. Після закриття ремонту перерахунок заборонений без окремого права адміністратора та запису в `audit_logs`.

### Податкові накладні, акти і дебіторка

- `tax_invoices`
- `receivables_control`

Статусу переносу на наступний період немає.

Правило:

```text
act_issued_at -> vat_status = ready_for_tax_invoice
```

Оплата не є умовою для реєстрації ПН. Якщо оплати немає, борг контролюється через `receivables_control`.

`tax_invoices` зберігає:

- дату події;
- дедлайн реєстрації;
- суму;
- ПДВ;
- статус;
- дату формування;
- дату відправки;
- дату реєстрації;
- квитанцію або відповідь шлюзу M.E.Doc / API ДПС.

`receivables_control` зберігає борг, дату боргу, дату видачі акта, дату підписання акта, відповідального менеджера і дату ескалації.

`payments.source_type` визначає, до чого належить платіж:

- `sale`
- `service_order`

Касова дисципліна ведеться через відкриття/закриття зміни і рухи каси.

Оплати підтримують:

- `cash`;
- `card_terminal`;
- `bank_transfer`;
- `mixed`.

`payments.transaction_number` зберігає номер транзакції, чека або банківського платежу.

Каса рахує окремо:

- готівку;
- картку через термінал;
- безготівку.

`terminal_transactions` підготовлена для другого етапу: інтеграції з платіжним терміналом. На першому етапі менеджер вручну вводить або фіксує номер транзакції.

### Файли, аудит, повідомлення

- `attachments`
- `document_templates`
- `generated_documents`
- `generated_document_versions`
- `audit_logs`
- `notifications`
- `client_notification_templates`
- `client_notifications`
- `repair_approvals`

`document_templates` зберігає шаблони друкованих форм: логотип, реквізити, HTML-шаблон, версію і автора.

`generated_documents` зберігає створені PDF-документи:

- квитанцію прийому;
- акт діагностики;
- заказ-наряд;
- акт виконаних робіт;
- акт видачі;
- товарний чек;
- накладну продажу;
- касовий чек;
- документ повернення;
- приходну накладну.

Кожен документ має номер, тип, статус, зв'язок із джерелом, PDF-шлях, версію, автора, підписанта і блокування після підпису.

`generated_document_versions` зберігає історію PDF-версій, щоб підписані документи не змінювалися заднім числом.

`client_notification_templates` зберігає тексти SMS, Telegram і Email для подій сервісного замовлення.

`client_notifications` зберігає журнал повідомлень клієнту:

- замовлення;
- клієнта;
- канал;
- подію;
- текст;
- статус;
- помилку провайдера;
- дату створення;
- дату відправки.

`repair_approvals` зберігає юридично важливе погодження ремонту:

- замовлення;
- одноразовий токен посилання;
- статус відповіді;
- телефон клієнта;
- знімок пристрою і несправності;
- знімок робіт;
- знімок запчастин;
- суму, яку відправили клієнту;
- строк;
- текст повідомлення;
- канал відправки;
- дату відправки;
- дату відповіді;
- IP і user-agent відповіді;
- коментар або ручну фіксацію менеджера.

Це важливо: після відправки погодження сума, роботи і запчастини в `repair_approvals` не повинні змінюватися заднім числом. Якщо пропозиція змінилася, створюється нове погодження.

Події:

- `device_received`;
- `diagnostics_done`;
- `repair_approval`;
- `waiting_part`;
- `ready_for_pickup`;
- `pickup_reminder`;
- `issued`.

Канали:

- `sms`;
- `telegram`;
- `email`.

`audit_logs` повинен отримувати записи для:

- створення документа;
- зміни статусу;
- прийому оплати;
- видачі товару;
- резерву;
- списання;
- входу/виходу користувача.

## Важливі backend-правила

1. Не дозволяти резерв, якщо `qty_available < qty`.
2. Не списувати товар при оплаті.
3. Списувати товар тільки при підтвердженій видачі або закритті ремонту.
4. Заборонити зміну закритих документів без окремого права.
5. Заборонити повернення без посилання на продаж або ремонт.
6. Кожну зміну залишку створювати через `stock_movements`.
7. Кожну критичну дію писати в `audit_logs`.
8. Фінансові дані віддавати тільки ролям із потрібними permissions.
9. Не дозволяти хаотичні переходи статусів ремонту.
10. Не дозволяти `closed`, якщо ремонт не був `issued`.
11. Не дозволяти `approval`, якщо немає результату діагностики.
12. Не дозволяти `waiting_parts`, якщо немає потреби в деталі.
13. Бухгалтер має бачити фінанси, закупки, склад, собівартість, ПДВ і звіти.
14. Бухгалтер не повинен змінювати технічні статуси ремонту або видаляти складські рухи.
15. Експорти мають підтримувати Excel і CSV; інтеграції 1С, BAS і M.E.Doc потрібно робити окремим backend-модулем.
16. Підписаний PDF-документ не можна редагувати, потрібно створювати нову версію.
17. Документ повернення повинен мати посилання на продаж, ремонт або платіж.
18. Квитанція прийому та акт виконаних робіт мають зберігатися як юридично важливі документи.
19. Сканер штрих-кодів працює як клавіатура: backend шукає товар по `barcode` або `sku`.
20. QR замовлення не повинен відкривати чужі дані без авторизації.
21. Оплата карткою не списує товар: списання тільки після видачі.
22. Повідомлення клієнту не повинні дублюватися без ручної дії менеджера.
23. Усі відправки SMS / Telegram / Email потрібно логувати з помилками провайдера.
24. Telegram-бот у другому етапі має відповідати клієнту по номеру замовлення, але тільки без розкриття зайвих персональних даних.
25. Погодження ремонту має фіксувати точну суму, роботи і запчастини на момент відправки клієнту.
26. Відповідь клієнта через посилання має записувати IP, час, результат і текст відправленого повідомлення.
27. Якщо клієнт відмовився, backend переводить ремонт у паузу або окремий статус відмови за правилами компанії.
28. Аналітика керівника повинна рахувати виручку, собівартість, прибуток, борги, повернення і списання з проведених документів.
29. Проблемні замовлення визначаються по часу в статусі, простроченій даті готовності і очікуванню запчастин.
30. Ефективність співробітників рахується окремо для менеджерів і інженерів.
31. Зарплату не можна рахувати по заказах у процесі.
32. Зарплата інженера рахується від `service_order_items`, де вказаний виконавець.
33. Для кожної послуги повинна бути схема `percent_of_work_amount` або `fixed_per_unit`.
34. У зарплату інженера не входять запчастини, витратні матеріали, доставка і додаткові витрати, якщо вони не оформлені окремою послугою.
35. Після закриття ремонту змінювати роботи, схему нарахування, ціну або виконавця можна тільки з окремим правом і audit log.
36. Статус переносу ПН на наступний період заборонений.
37. Акт видано клієнту означає податкову подію в поточному періоді.
38. Одиницю зі статусом `on_shelf` не можна залишити без координатної комірки.
39. Сканування QR/штрихкоду має відкривати конкретну одиницю або заказ.
40. Кожен рух одиниці по цеху має писатися в `order_unit_movements`.
41. Статус `cannot_repair` потребує коментаря і не повинен входити в активні ремонти.

## Аналітичні запити для backend

Для продуктивної роботи Dashboard бажано підготувати materialized views або кешовані API:

- виручка за день / місяць;
- прибуток за період;
- продажі й ремонти по днях;
- борги клієнтів;
- закази по статусах;
- проблемні закази без руху;
- ефективність менеджерів;
- ефективність інженерів;
- мінімальні залишки;
- оборачиваемість товарів;
- повернення і списання.

Формула прибутку:

```text
дохід - собівартість = прибуток
```
